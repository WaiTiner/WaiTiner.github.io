了解java基本关键字

static

被static修饰的成员变量和成员方法是独立于该类的，它不依赖于某个特定的实例变量，也就是说它被该类的所有实例共享。所有实例的引用都指向同一个地方，任何一个实例对其的修改都会导致其他实例的变化。

无论是变量，方法，还是代码块，只要用static修饰，就是在类被加载时就已经”准备好了”,也就是可以被使用或者已经被执行，都可以脱离对象而执行。反之，如果没有static，则必须要依赖于对象实例。

静态变量和实例变量的区别：静态变量在类加载的时候就存在了，它在jvm的内存中只存在一次，修改了会造成永久改变，而实例变量跟实例共存亡，没产生一个实例，就会产生一个实例变量。

静态方法：同在类加载的时候就存在了，所有它可以通过类名直接调用，并且必须实现，不能是抽象的。

静态代码块：可以存在于任何地方

缺点：不能被this，super调用，同时只能调用static方法和变量，且必须要进行初始化

final

修饰类：不能被继承，此时该类中的所有方法都会被隐式指定为final方法
修饰方法：不能被重写(覆盖)
修饰变量：包括成员变量和局部变量，此时变量表示常量，不可以被修改，当修饰基本数据类型时，值不可以被修改，修饰引用数据类型时，地址不可以被修改，值可以发生变化。

熟悉java容器

了解部分集合类如：ArrayList，LinkedList，HashMap，ConcurrentHashMap 。

ArrayList

  1. 是一种数组列表的数据结构
  2. 访问速度极快，适合做读取
  3. 自动扩容，插入或删除元素时，会复制后面的元素进行前后移动完成操作，消耗大，前后操作不影响
  4. 线程不安全
  5. 允许重复，允许所有值

  6. 先进先出

    用法https://www.cnblogs.com/lxqiaoyixuan/p/7156936.html
    场景https://www.cnblogs.com/itdragon/p/8400461.html

LinkedList

  1. 链表的数据结构
  2. 适合插入，删除频繁时使用，适合首位操作
  3. 线程不安全
  4. 允许重复，有序

HashMap

  1. 类似数组链表的数据结构
  2. 允许null键和null值，当存入重复键时，会有返回值，为被覆盖之间的值，无重复键，则返回null。
  3. 效率高，无序，不重复，无序表现为，取出键值对的输出顺序和键的大小排序有关，和输入顺序无关
  4. 线程不安全，不能够在多线程环境下使用

HashMap的主干是一个Entry数组，每个entry数组存储一个key-value键值对。在存储的过程中，先将key通过hashCode（）再通过hash（）得到当前元素在数组中的下标，然后如果发现该下标下为null则直接存储到这个位置；如果有entry存在了，则要先判断key是否相同，如果相同，则覆盖原来的value，如果不同，则将当前的key-value以链表的形式放入当前下标

ConcurrentHashMap

  1. 支持并发操作，分段锁结构
  2. 默认一个ConcurrentHashMap里有16个HashMap
  3. 是一个线程安全的并发容器
  4. concurrentHashMap的操作是一个组合，先get再put，所以多个线程的操作出现了覆盖。

线程协作

当多个线程共同去完成某个事情的时候，如果需要某些部分在其他部分之前完成，则需要线程之间的协调。

  1. join方法：若是在a线程中调用了b线程的join方法b.join()则a线程会先让b线程运行结束后再运行自己。

  2. wait方法：

    当a线程在方法中调用了wait()方法，则需要其他线程用notify()或notifyAll()方法唤醒线程执行。

且这三个方法都属于Object，只能在同步代码块或同步方法中调用。

且使用wait方法期间，线程会释放锁，不然其他线程的无法执行notify()或notifyAll()来唤醒线程，造成死锁。

3.在J.U.C类库下使用的是Condition类的 await(),signal(),signalAll(),实现方法2的操作，不同的是这里的await在使用的时候可以指定await的条件。

JVM内存分布

作一个JVM运行时的内存区域 如下：

JVM内存区域{
        JVM初始运行的时候都会分配好Method Area（方法区）和Heap（堆）；

        而JVM 每遇到一个线程，就为其分配一个Program Counter Register（程序计数器）, VM Stack（虚拟机栈）和Native Method Stack （本地方法栈）；

        当线程终止时，三者（虚拟机栈，本地方法栈和程序计数器）所占用的内存空间也会被释放掉。这也是为什么我把内存区域分为线程共享和非线程共享的原因；

        线程共享{
                heap（堆){
                        复杂；
                        保存对象实例的属性值，属性的类型；
                        和类型标记等不保存对象的方法；
                        属于垃圾回收的主要场所；
                        通过栈中的四字节内存地址寻找堆中的对象实例；
                        物理不连续，逻辑连续的内存空间；
                        可动态增加其内存，增加失败会抛出 OutOfMemoryError 异常；
                        Young Generation（新生代）{}
                        Old Generation（老年代）{}
                }
                metod area（方法区）{
                        用于存放被加载的类信息，常量，静态变量，JIT编译后的代码；
                        Runtime Constant Pool（运行时常量池rcp）{
                            Class 文件中的常量池（编译器生成的各种字面量和符号引用）会在类加载后被放入这个区域；
                            允许动态生成常量，如String类中的intern();
                        }
                }
        }
        非线程共享{
                 VM Stack（虚拟机栈）{
                                保存简单类型；
                                保存对象类型的地址；
                                栈内存的生命周期随着线程共生死；
                                在虚拟机栈中方法执行对应着栈帧的进出；
                                Frame（栈帧）{
                                    遵循先进后出原则；
                                    Local Variable Array（局部变量表）{}
                                    Openrand Stack（操作数栈）{}
                                    ReferencetoConstant     
                                    Poll（常量池引用）{}
                                }
                 }
                 Native Method Stack（本地方法栈）{为本地方法服务的栈区}
                 Program Counter Register（程序计数器）{
                                字节码的位置指示器
                 }
        }
}

垃圾收集机制
主要针对堆和方法区收集

判断算法：

判断是否可以被回收
. 引入计数算法

给对象增加一个计数器，对象增加了一个引用时+1，对象失去一个引用时-1，当计数为0后，计入回收，但由于循环引用的存在，JVM不使用引入计数算法

. 可达性分析算法

通过GC-roots作为起始点，进行搜索，可达到的对象标记存活，不可达到的对象标记回收。

现代商用虚拟机的收集算法：

采用分代收集算法，根据不同对象的生命周期将内存分为几个块，不同的块采用不同的收集算法

将堆分为新生代和老年代，新生代采用复制算法收集，首先将新生代的内存区域划分为Eden区和两块Survivor区，比例为8：1，这时，平时新生代使用时会用一块Eden 区和一块Survivor区，等到回收的时候将这两块区域的存活对象复制到另一块Survivor区，然后清理Eden和Survivor区。如果最后一块Survivor区不够用，可以到老年代分担。
老年代采用标记整理，或者标记清除算法。老年代的对象存在时间长，如大对象：很长的字符串或数组，将直接保存到老年代。

Minor GC：当Eden满了后就会触发；

Full GC ：

  • 调用System.gc()的时候会建议JVM执行；
  • 老年代空间不足；
  • 老年代分配担保失败；

对方法区的回收主要时针对运行时常量池的回收和类的卸载
类的卸载条件：

  1. 该类的所有对象都被回收，也就是说堆中不存在该类的实例。
  2. 加载该类的classLoader已经被回收。
  3. 该类的class对象没有在其他地方被引用。

类的生命周期
加载，验证，准备，解析，初始化
类的加载不会在编译期一次性执行，会在运行期首次动态加载，为了保证运行效率。

加载解释：
类的加载由类加载器完成，可以使用JVM自带的启动类加载器，或者外部加载器，也可以程序员自定义类加载器，通常自定义的类加载器是为了保证安全性，在java中类加载器的使用要求符合双亲委派模型的原则。

关于双亲委派模型，它保证了java类的加载伴随着类加载器的调用顺序产生一种优先层次关系，保证了运行的安全性，防止不可信类扮演可信类。
它的原理：除了顶层的启动类加载器外，其他的类加载器都需要有它的父类加载器，当一个类加载器收到加载请求的时候它会交由它的父类加载器执行。

余下过程的重点：

通过一个全限定名（import+…）来获取定义此类的二进制字节流(jar包，动态代理)；
通过在内存中生成的此类class对象，作为访问此类在方法区各种数据的入口；
在类变量的准备阶段，当修饰符为static时，会为它在方法区配置内存，并为变量配置初始值，同时要注意如果同时被final修饰即变为常量则不会为其配置初始值，而是按表达式来初始化；

类的初始化：真正开始执行程序中的java代码，为虚拟机执行类构造器()的过程，同时会保证父类的()先于子类执行，而且与实例构造器()不同的是，不需要显示调用父类的构造器，这样也就说明了父类的静态语句块会先于子类。

面向对象思想

封装，继承，多态

  1. 通过抽象的数据类型，将具体实现封装在程序里面，使其不需要了解内部实现，只需要调用接口即可调用功能，减少了耦合，提高了系统的复用度，降低了大型系统的风险性，并且能减少维护的负担。
  2. 继承能够使子类获得父类的所有非private方法和属性，并且子类能够替换父类的所有方法，同时父类对象指向子类引用是一种向上转型。
  3. 多态分指编译时多态和运行时多态，在运行时多态中：对于A a = new B(); 编译看左边，即对于父类与子类的关系中，该写法在a编译成功，则运行，不过这时是按照子类b的方法执行，返回b的结果。
    而在编译时多态中，指方法的重载

java设计模式
单例模式
singleton：确保一个类只有一个实例，并提供它的全局访问点
懒汉式：

public class Singleton{

        private static Singleton uniqueInstance;

        private singleton(){}

        public static synchronized Singleton getUniqueInstance(){
                if(uniqueInstance == null){
                        uniqueInstance = new Singleton();
                }
                return uniqueInstance;
        }
}

使用一个私有静态变量，一个私有构造函数，一个公有静态且加锁的函数，来保证只有通过公有函数才能返回唯一的实例，这里用synchronized给此函数加锁，确保了线程安全，但是其他线程进入需要等待，一定程度上造成了性能的损耗，但也避免了多次实例化，节约了资源。

避免造成性能损耗的线程安全懒汉模式—–双重校验锁

public class Singleton{

        private static uniqueInstance;

        private Singleton(){}

        public static Singleton getUniqueInstance(){
            if(uniqueInstance == null){
                    synchronized(Singleton.class){
                            if(uniqueInstance == null){
                                    uniqueInstance = new Singleton();
                            }
                    }
            }
            return uniqueInstance;
        }

}

此处判断了是否uniqueInstance为空，为空才上锁，避免了每次都要等待，此处一定要再次判断uniqueInstance是否为空，因为上一次判空相当于去除掉了避免多次实例化的实现条件。

单例模式最佳实现：枚举
好处是避免了在复制序列化和反射攻击下的多次实例化

数据库

了解事务特性ACID

事务是指一组满足ACID特性的操作
Atomicity 原子性：指事务时不可分割的，要么全部成功，要么全部失败回滚。
Consistency 一致性：所有事务对一个数据的读取在事务前后都保持一致。
Isolation 隔离性：一个事务对数据作的修改在事务完成之前对其他事务是不可见的。
Durability 持久性：事务对数据进行的修改在commit之后会永远保存在数据库中不可改变。

事务特性的关系式如下：（它们并不是简单的平级关系）

执行结果正确{
    一致性{
        原子性{并发/不并发}；
        隔离性{并发}；
    }
}
对应数据库崩溃{
    持久性；
}

了解并发一致性问题
脏读，不可重复读，幻影读，修改丢失
分别对应：
t1修改了数据，然后回滚了，但是当t2读这个数据的时候读到了t1修改后的数据；
t2读取一个数据，t1对其进行了修改，t2再读就读到了和之前不一样的数据；
t1读取一个范围内的数据，t2在这个范围内插入了数据，t1再读取读到了和第一次读取不一样的数据；
t1对数据进行了修改，但是t2也对数据进行了修改，这样t1对数据的修改被覆盖了；
